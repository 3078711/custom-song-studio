# 深度编辑功能：Suno API 能力整合方案

> **评估维度**：操作性、扩展性、前瞻性

---

## 〇、能否满足「用户定制歌曲创作与编辑」要求？

### 需求覆盖矩阵

| 用户需求场景 | 对应能力 | 是否覆盖 |
|--------------|----------|----------|
| 对整体满意，想加民族特色/换人声 | 上传并翻唱 | ✅ |
| 歌曲太短，想拉长 | 延长音乐、上传并扩展 | ✅ |
| 某一段不满意，想重做 | 替换音乐分区 | ✅ |
| 需要伴奏做卡拉OK/翻唱 | 人声分离 | ✅ |
| 只有人声，想加伴奏 | 添加伴奏 | ✅ |
| 只有伴奏，想加人声 | 添加人声 | ✅ |
| 需要固定人声风格 | 生成 Persona | ✅ |
| 两首歌混在一起 | 生成混音 | ✅ |
| 需要分轨/MIDI 做后期 | 人声分离、生成 MIDI | ✅ |
| 需要 MV | 创建音乐视频 | ✅ |
| 风格描述不够好 | 提升音乐风格 | ✅ |

### 结论

- **创作**：现有工作台已有；风格提升可增强创作质量
- **编辑**：覆盖「换风格、拉长、局部重做、分轨、加人声/伴奏、Persona、混音、MIDI、MV」等主流场景
- **操作性**：通过引导、预设、流程提示提升
- **扩展性**：能力注册机制支持快速接入新 API
- **前瞻性**：版本溯源、流程编排、多 AI 接入已预留

## 一、Suno API 能力清单与输入要求

基于 [Suno API 文档](https://docs.sunoapi.org/cn) 与 [llms.txt 索引](https://docs.sunoapi.org/llms.txt)，整理如下：

| 功能 | API 端点 | 输入要求 | 输出 | 平台数据是否满足 |
|------|----------|----------|------|------------------|
| **延长音乐** | `/api/v1/generate/extend` | audioId | 更长音轨 | ✅ 版本有 sunoAudioId |
| **替换音乐分区** | `/api/v1/generate/replace-section` | taskId, audioId, infillStartS, infillEndS, prompt, tags, title | 替换后的完整音轨 | ✅ 版本有 sunoTaskId, sunoAudioId |
| **人声与乐器分离** | `/api/v1/vocal-removal/generate` | taskId, audioId, type(separate_vocal/split_stem) | 人声/伴奏/分轨 URL | ✅ 同上 |
| **上传并翻唱** | `/api/v1/generate/upload-cover` | uploadUrl, style, prompt, title | 新风格音轨 | ⚠️ 需公网 audioUrl |
| **上传并扩展** | `/api/v1/generate/upload-extend` | uploadUrl, continueAt, style, prompt, title | 更长音轨 | ⚠️ 需公网 audioUrl |
| **添加人声** | `/api/v1/generate/add-vocals` | uploadUrl(伴奏), prompt, style, title | 带人声完整曲 | ⚠️ 需公网 audioUrl |
| **添加伴奏** | `/api/v1/generate/add-instrumental` | uploadUrl(人声), tags, title | 带伴奏完整曲 | ⚠️ 需公网 audioUrl |
| **生成混音** | `/api/v1/generate/mashup` | 两个音频 | 混音曲 | ⚠️ 需查具体参数 |
| **从音频生成 MIDI** | 需分离后的分轨 | 分轨 URL | MIDI 文件 | ⚠️ 依赖人声分离 |
| **创建音乐视频** | 需查 | 音轨 | MP4 视频 | ⚠️ 需查具体参数 |
| **生成 Persona** | 基于已生成音乐 | taskId/audioId | personaId | ✅ 版本有 sunoTaskId |
| **提升音乐风格** | `/api/v1/style/generate` | content(风格描述) | 优化后的风格文本 | 无需音频，属创作辅助 |

**说明**：
- `uploadUrl`：Suno 需从公网拉取音频，本地路径 `/api/media/audio/xxx` 需部署后具备公网域名才可用
- 基于 `taskId`/`audioId` 的接口：仅适用于 **Suno 平台生成的音频**，我们保存了这些 ID 的版本可直接使用

## 三、功能放置策略

### 3.1 核心原则

1. **基于版本操作**：深度编辑均以「选中版本」为输入
2. **入口集中**：避免分散在多个页面
3. **分阶段实施**：先做输入要求简单、价值高的功能

### 3.2 推荐方案：新建「深度编辑」模块

**不建议**全部塞进版本卡片，会导致：
- 版本卡片过于拥挤
- 每个功能参数不同，需要独立表单

**建议**：在项目工作台内新增 **「深度编辑」** 区域，作为版本列表的扩展操作区。

---

## 四、具体布局方案

### 4.1 方案 A：工作台内嵌「深度编辑」面板（推荐）

**位置**：`/dashboard/projects/[id]/workbench`

**交互**：
1. 用户选中某个版本后，版本卡片下方或右侧展开「深度编辑」面板
2. 面板内为功能入口网格/列表，如：
   - 上传并翻唱
   - 延长音乐
   - 替换分区
   - 人声分离
   - 添加人声
   - 添加伴奏
   - 生成 Persona
   - 创建音乐视频
   - 等
3. 点击某项进入对应表单（弹窗或内嵌表单）
4. 提交后创建「编辑任务」，与生成任务类似，支持轮询与结果展示

**优点**：不离开工作台，操作路径短  
**缺点**：工作台布局更复杂，需合理折叠/展开

---

### 4.2 方案 B：独立「深度编辑」页面

**路由**：`/dashboard/projects/[id]/studio` 或 `workbench/studio`

**布局**：
- 左：版本列表（选择要编辑的版本）
- 中：功能选择 + 参数表单
- 右：任务进度与结果预览

**优点**：空间充足，适合复杂表单  
**缺点**：多一层导航，与创作工作台割裂

---

### 4.3 方案 C：版本卡片「更多」下拉菜单

**位置**：版本卡片操作区增加「更多」或「深度编辑」

**交互**：下拉列出所有深度编辑功能，点击打开弹窗表单

**优点**：实现简单，入口明确  
**缺点**：弹窗空间有限，复杂功能体验一般

## 六、公网 URL 解决方案

**问题**：Cover、Extend(上传)、Add Vocals、Add Instrumental 需要 `uploadUrl`，Suno 需能访问。

**可选方案**：

1. **部署后使用站点域名**：`https://你的域名/api/media/audio/xxx.mp3`，Suno 可访问
2. **临时上传到对象存储**：生成时把音频上传到 OSS/S3，获取临时公网 URL 传给 Suno
3. **Suno 文件上传 API**：若文档提供 Base64/URL 上传接口，可先上传再获得 Suno 侧 URL

## 十、操作性增强设计

### 10.1 操作引导与能力说明

| 问题 | 设计 |
|------|------|
| 用户不清楚「延长」与「扩展」区别 | 每个能力入口提供简短说明 + 示例场景 |
| 替换分区需填时间，用户不知如何选 | 提供音频波形/时间轴预览，支持拖拽选择区间；或「按段落」选择（Verse1/Chorus） |
| 人声分离后不知道能做什么 | 分离完成后展示「可基于此伴奏：添加人声、上传翻唱」等后续操作入口 |

### 10.2 一键预设

| 预设 | 对应能力 | 参数 |
|------|----------|------|
| 藏族风格翻唱 | 上传并翻唱 | style: Tibetan folk, Himalayan vocal quality... |
| 蒙古族风格翻唱 | 上传并翻唱 | style: Mongolian throat singing... |
| 卡拉OK 伴奏 | 人声分离 | type: separate_vocal → 输出伴奏 |
| 扩展 30 秒 | 延长音乐 | 从末尾续写 |

### 10.3 流程提示

- 在「添加人声」入口提示：「需纯伴奏，可先做人声分离」
- 在「添加伴奏」入口提示：「需人声或旋律轨，可先做人声分离」
- 在「上传翻唱」入口提示：「需公网可访问的音频 URL」

## 十一、扩展性架构设计

### 11.1 编辑能力注册表

```ts
// 编辑能力类型：统一注册，便于新增
interface EditCapability {
  id: string;
  label: string;
  description: string;
  category: "transform" | "extend" | "split" | "combine" | "export";
  inputType: "suno_audio" | "upload_url" | "multi_audio";
  paramsSchema: Record<string, ParamDef>;
  apiHandler: (params: unknown) => Promise<string>; // 返回 taskId
  resultHandler?: (taskId: string, raw: unknown) => VersionCreateInput;
}
```

- **新增能力**：在注册表中添加一项即可，无需改业务逻辑
- **参数驱动**：表单根据 `paramsSchema` 自动生成，减少重复代码

### 11.2 任务与能力解耦

- **EditTask** 统一结构：`{ id, capabilityId, sourceVersionId, params, status, resultVersionId?, ... }`
- 回调根据 `capabilityId` 路由到对应 `resultHandler`，不写死 if-else

### 11.3 多源输入扩展

| 输入类型 | 当前 | 未来 |
|----------|------|------|
| 平台生成版本 | ✅ 有 sunoTaskId/audioId | ✅ |
| 持久化音频 | ⚠️ 需公网 URL | 支持上传到 OSS 获取 URL |
| 用户上传 | 未支持 | 支持上传干声/参考曲作为编辑源 |
| 人声分离产物 | 未支持 | 分离结果可保存为「分轨版本」，作为后续编辑输入 |

## 十二、前瞻性设计

### 12.1 版本溯源

- **sourceVersionId**：编辑结果版本记录「来源版本」
- **编辑链**：A → B(翻唱) → C(扩展)，可展示「版本树」或「编辑历史」
- **用途**：追溯、对比、回滚、客户沟通

### 12.2 流程编排（预留）

- 短期：单步编辑为主
- 中期：支持「人声分离 → 添加伴奏」等 2 步流程，中间结果可暂存
- 长期：可视化编排（如「分离 → 换人声 → 混音」），可做成独立功能

### 12.3 多 AI 能力接入（预留）

- 抽象层：`MusicEditProvider` 接口，Suno 为具体实现
- 未来若有 Udio、Synthesizer V、本地 RVC 等，可新增 Provider 而不改动核心流程
- 同一能力（如「更换人声」）可由不同 Provider 实现，用户可选

### 12.4 数据模型预留字段

```ts
// SongVersion 扩展
interface SongVersion {
  // ... 现有字段
  sourceVersionId?: string;   // 来源版本（编辑时）
  editCapabilityId?: string;  // 由何种编辑能力产生
  stems?: {                  // 人声分离产物（可选持久化）
    vocal?: string;
    instrumental?: string;
    [key: string]: string | undefined;
  };
}
```

### 12.5 其他前瞻

| 方向 | 说明 |
|------|------|
| 成本可见 | 展示各能力消耗积分、预估耗时 |
| 批量操作 | 对多版本批量「人声分离」等 |
| 客户协作 | 编辑结果可发客户审批、反馈 |
| 导出格式 | WAV、分轨、MIDI 等导出 |
